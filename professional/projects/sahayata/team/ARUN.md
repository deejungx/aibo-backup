## 📍 Role Details

### **1. Customer Signal Lead (Weekly)**

**Weekly Rhythm:**
- **Mon-Fri:** Conduct 3-5 customer discovery visits
- **Friday:** Prepare signals + interview domain experts (1 hour)
- **Monday 9 AM:** Present 3 customer signals to team (5 min)

**What Arun Brings to Planning:**
Each signal includes:
- **Severity:** 1-5 scale (how urgent? how many customers?)
- **Quote:** What customers actually said (paraphrase is fine)
- **Impact:** What changes for them if we fix this?
- **Blocker?:** Is this blocking them from using the product?

**Example Signal:**
```
SIGNAL #1: Prompt templating is too rigid
- Severity: 4/5 (3 customers mentioned, came up in demos)
- Quote: "We want to swap out parts of the prompt for different use cases, but the whole template changes every time"
- Impact: Customers could run multiple variants. Right now they have to modify code.
- Blocker: No, but it's slowing down their experimentation
```

**How Team Uses It:**
- Sprint backlog gets labeled Red/Yellow/Green
- 🔴 Red = Solves customer signal (high priority)
- 🟡 Yellow = Supports sprint goal (medium)
- 🟢 Green = Nice-to-have/tech debt (low)
- Sprint goal becomes customer-driven: *"Ship flexible prompts + detailed errors (Signals #1 & #2)"*

**Why It Matters:**
- Removes backlog guesswork
- Team knows *why* they're building (customer voice)
- Work is measurable (did this signal actually help?)
- Reduces meetings (customers already decided priorities)

### **2. Knowledge Librarian (Weekly)**

**Weekly Maintenance (1 hour Friday):**

**What Arun Documents:**

1. **Sources** — Where customers find info
   - Internal: Knowledge bases, past emails, spreadsheets, team expertise
   - External: Websites, government sites, forums, news
   - Frequency: Daily/weekly/monthly patterns

2. **Tools** — Workflow stack they use daily
   - Communication: Email, WhatsApp, Slack, Phone
   - Data/Documents: Google Sheets, CRM, Notion
   - Integrations they wish existed

3. **Intents** — What they're *really* trying to accomplish (not just keywords)
   - Example: "Qualification (60%)" = customers want to know if they fit, don't waste time if not
   - Team uses this to build agents that respond differently by intent

4. **Workflows** — Step-by-step how they handle scenarios
   - Full process: Inquiry → Qualify → Send Checklist → Collect Docs → Decide
   - Which steps are manual (automation opportunity)
   - Time spent per step (ROI calculation)

5. **FAQs** — What they get asked repeatedly
   - Top questions by frequency
   - Current answer process
   - Time spent per answer → automation priority

6. **Domain Expertise** — Institutional knowledge only staff know
   - "Tech background is highly valued here"
   - "Processing times just changed to 8 weeks"
   - "Students from India study at night → suggest schedules accordingly"
   - Differentiates agent from generic chatbot

**How This Becomes Agent Training:**

Library → Agent Brief:
```
AGENT BRIEF: Education Consultancy Chatbot

INTENTS TO AUTOMATE: Qualification (60%), Timeline (25%)
DATA SOURCES: Past emails (vector DB), requirements spreadsheet, policy docs
TOOL INTEGRATIONS: WhatsApp, Gmail, Google Sheets, CRM
DOMAIN KNOWLEDGE: Tech background valued, 8-week processing, students from India study night
WORKFLOWS TO SUPPORT: Steps 1-3 (full automation), 4-5 (template-based)
```

## 💡 Key Principles

- **Customer-driven, not speculation:** Only document what customers actually said/showed you
- **Specific, not vague:** "Prompts too rigid" beats "prompts bad"
- **Tied to outcomes:** "They'll experiment 10x faster" not "theoretically better"
- **Living document:** Library updates weekly as customer needs shift
- **Knowledge capture:** Prevent institutional knowledge from leaving when people do
